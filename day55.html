Concept

What is Recursion?

Recursion is when a function calls itself to solve a smaller version of the same problem.

The Recipe

def solve(problem):
    if problem is tiny:        # ← base case
        return known_answer
    else:
        smaller = shrink(problem)
        return solve(smaller)     # ← recursive call

Analogy: Matryoshka Dolls

To open the biggest doll, open the next size down … until you reach the tiny solid doll (base case). Then stack them back up.

Every recursive function needs:

  • Base case — stops the recursion
  • Recursive case — calls itself on a smaller input

Walkthrough

Countdown — Your First Recursive Function

def countdown(n):
    if n <= 0:              # ← base case
        print("Go!")
        return
    print(n)                 # ← do something
    countdown(n - 1)        # ← recursive call

countdown(3)

Trace for countdown(3)

countdown(0) → "Go!" ↩
countdown(1) → prints 1 ↩
countdown(2) → prints 2 ↩
countdown(3) → prints 3 ↩
Call stack (top = current)
▼ Output: 3 2 1 Go! — each call prints, then recurses deeper.

📚 The Call Stack

How Python remembers where to return after each recursive call

Concept

How the Call Stack Works

Each function call pushes a stack frame onto the call stack containing local variables and the return address. When a function returns, its frame is popped off.

Factorial: fact(4)

def fact(n):
    if n <= 1:
        return 1          # base
    return n * fact(n - 1)  # recursive

Stack Trace

fact(1) → 1 ↩
fact(2) → 2*1 = 2 ↩
fact(3) → 3*2 = 6 ↩
fact(4) → 4*6 = 24 ↩

Frame returns a value → caller multiplies → returns to its caller

☠ Stack Overflow: No base case → infinite recursion → RecursionError: maximum recursion depth exceeded (Python caps at ~1000 frames).

Walkthrough

Factorial — n!

Definition: n! = n × (n−1) × (n−2) × … × 1  |  Base: 0! = 1, 1! = 1

def factorial(n):
    # Base case: 0! and 1! = 1
    if n <= 1:
        return 1
    # Recursive: n * (n-1)!
    return n * factorial(n - 1)

# Try it:
print(factorial(5))  # → 120

How it unfolds:

factorial(5)
= 5 * factorial(4)
= 5 * 4 * factorial(3)
= 5 * 4 * 3 * factorial(2)
= 5 * 4 * 3 * 2 * factorial(1)
= 5 * 4 * 3 * 2 * 1 # base!
= 120
✅ Key insight: The multiplications are deferred until the base case is reached. Then they unwind right-to-left ("backward").

Walkthrough

Fibonacci Sequence

Definition: F(0)=0, F(1)=1, F(n)=F(n−1)+F(n−2) — each number is the sum of the two before it.

def fib(n):
    if n <= 1:
        return n         # F(0)=0, F(1)=1
    return (fib(n - 1) +
            fib(n - 2))

print(fib(6))  # → 8

Sequence: 0,1,1,2,3,5,8,13,21…

Recursion Tree for fib(5)

fib(5) ├── fib(4) │ ├── fib(3) │ │ ├── fib(2) │ │ │ ├── fib(1)=1 │ │ │ └── fib(0)=0 │ │ └── fib(1)=1 │ └── fib(2) │ ├── fib(1)=1 │ └── fib(0)=0 └── fib(3) [same as above]
⚠ Performance trap: Naive fib(35) makes ~18M calls! Each call spawns 2 more → O(2n). Memoization fixes this (next slide).

Comparison

Recursion vs Iteration

🌀 Recursion

  • Elegant for tree/divide-and-conquer problems
  • Uses call stack automatically
  • Risk of stack overflow for deep inputs
  • Often more readable for self-similar problems
  • Can be slower due to function call overhead

⭯ Iteration

  • Always O(1) extra space (no stack frames)
  • Faster in practice (no call overhead)
  • Can be harder to write for tree/graph problems
  • Never hits recursion depth limit
  • Easier to debug (no stack unwinding to trace)
💡 Rule of thumb: Use recursion when the problem is naturally self-similar (trees, divide-and-conquer). Use iteration when performance matters or depth could exceed ~1000.

Solutions

Check Your Work

# Ex 1: sum_n
def sum_n(n):
    if n == 0:
        return 0
    return n + sum_n(n - 1)

# Ex 2: reverse
def reverse(s):
    if len(s) <= 1:
        return s
    return s[-1] + reverse(s[:-1])
# Ex 3: power
def power(b, e):
    if e == 0:
        return 1
    return b * power(b, e - 1)

# Ex 4: palindrome
def is_pal(s):
    if len(s) <= 1:
        return True
    if s[0] != s[-1]:
        return False
    return is_pal(s[1:-1])

Summary

Key Takeaways

1
Base case first. Every recursive function must have a condition that stops the recursion. Without it, you get stack overflow.
2
The call stack tracks your place. Each recursive call pushes a frame; each return pops one. Visualize the stack to debug recursion.
3
Deferred operations unwind in reverse. Work done after the recursive call (like n * fact(n-1)) waits until the base case returns.
4
Iteration is not always better. Use recursion when the problem is self-similar (trees, divide-and-conquer). Use iteration for performance.
