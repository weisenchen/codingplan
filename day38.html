🧬 What Is Inheritance?

Inheritance lets a child class reuse and extend the behavior of a parent class.

🔨

Analogy: A power drill inherits from a regular drill. It has all the same features plus extra ones. You don't reinvent the chuck — you add the motor.

Task ← parent (general)
┣━ PriorityTask ← child (more specific)
┗━ DeadlineTask ← another child
Children inherit all methods & attributes from the parent 
class Task:
    def __init__(self, title):
        self.title = title
        self.done = False
 
class PriorityTask(Task):       # (Task) = "inherits from Task"
    pass                              # has .title, .done for free

Syntax: class Child(Parent): — put the parent name in parentheses.

Inheritance is the "is-a" relationship. A PriorityTask IS a Task, plus more. Emphasise that the child doesn't need to re-implement anything it doesn't want to change.

⬆️ super() — Calling the Parent

When a child overrides __init__, it usually still needs the parent's setup. Use super() to call it.

class PriorityTask(Task):
    def __init__(self, title, priority):
        super().__init__(title)   # ← runs Task.__init__
        self.priority = priority    # ← new attribute
 
t = PriorityTask("Write report", "high")
print(t.title)     # inherited from Task
print(t.priority)  # new in PriorityTask

💡 Always call super().__init__() when overriding the constructor — otherwise the parent's attributes never get created.

def mark_done(self):
    super().mark_done()          # reuse existing logic
    print(f"Priority task '{self.title}' done!")

super() returns a proxy object that delegates method calls to the parent class. It's not a function call that "goes up" — it's a nice API for cooperative multiple inheritance. For single inheritance, just think of it as "the parent's version of this method".

🔮 Dunder Methods — What Are They?

"Dunder" = Double Underscore. These are Python's protocol methods — they let your objects hook into built-in operations like print(), ==, len(), and more.

MethodTriggered ByPurpose
__init__ Task("x") Constructor — set up new objects
__str__ print(obj), str(obj) Human-readable, informal description
__repr__ repr(obj), debug console Unambiguous, developer-focused repr
__eq__ obj1 == obj2 Equality comparison
__lt__ obj1 < obj2 Less-than (for sorting)
🎭

Rule of thumb: __repr__ is for developers (should recreate the object). __str__ is for users (should be readable). If you only define one, define __repr__ — Python falls back to it for __str__.

Dunder methods are what make Python classes "feel native" — they integrate with print(), ==, sorting, iteration, etc. Without __str__, print(obj) shows something like '<__main__.Task object at 0x7f...>'.

🖨️ __str__ vs __repr__ — Side by Side

class Task:
    def __init__(self, title):
        self.title = title
        self.done = False
 
    def __repr__(self):                    # dev-friendly
        return f"Task('{self.title}')"    # enough info to recreate
 
    def __str__(self):                     # user-friendly
        status = "✓" if self.done else "○"
        return f"{status} {self.title}"
>>> t = Task("Buy groceries")
>>> repr(t)
"Task('Buy groceries')"
>>> print(t)
○ Buy groceries
>>> t.done = True
>>> print(t)
✓ Buy groceries

Show how each is triggered. The repr should ideally return a string that, when passed to eval(), recreates the object. The str is what users see in print().

⚖️ __eq__ — Comparing Objects by Value

Without __eq__, == checks memory identity — two objects with identical data are not equal:

t1 = Task("Buy milk")
t2 = Task("Buy milk")
print(t1 == t2)    # False ← same content, different objects!

Define __eq__ to compare by value instead:

def __eq__(self, other):
    if not isinstance(other, Task):     # guard: wrong type
        return NotImplemented           # let Python handle it
    return (self.title == other.title
            and self.done == other.done)
>>> t1 = Task("Buy milk")
>>> t2 = Task("Buy milk")
>>> t1 == t2
True

⚠️ Always check the type of other with isinstance(). Return NotImplemented (not False) for wrong types so Python can try the reverse comparison.

Without __eq__, Python compares by id() — two separate objects are never ==. This is a common gotcha. __eq__ is also used implicitly by set() and dict() for uniqueness, though for those you also need __hash__.

🛠️ Walkthrough: Equality Across Inheritance

Should two PriorityTasks with the same title, done status, and priority be equal? Let's add it.

class PriorityTask(Task):
    # ... __init__, __repr__, __str__ as above ...
 
    def __eq__(self, other):
        if not isinstance(other, PriorityTask):
            return NotImplemented
        # Call parent __eq__ for title + done, then add priority
        return super().__eq__(other) and self.priority == other.priority
>>> a = PriorityTask("Fix bug", "high")
>>> b = PriorityTask("Fix bug", "high")
>>> a == b
True
>>> c = PriorityTask("Fix bug", "low")
>>> a == c
False

💡 Notice how we reused super().__eq__() — the parent already knows how to compare title and done. We only add the priority check.

This is a great demonstration of DRY through inheritance — the parent has the base equality logic, the child extends it. This is exactly how real codebases work.

⚡ is vs == — Don't Confuse Them

== — Value Equality

Checks if two objects have the same content.

print(t1 == t2)   # True — same task

Can be customized with __eq__.

is — Identity

Checks if two variables point to the exact same object.

print(t1 is t2)   # False — diff objects
print(t1 is t1)   # True — same reference

Cannot be overridden (Python internal).

⚠️ Common pitfall: t1 is None is correct (identity with singleton). t1 == None works but is non-idiomatic. Use is for None, True, False.

This is one of the most common Python interview questions. is = identity (id()), == = value (calls __eq__).

🏋️ Exercise: Build a DeadlineTask

Your Turn

Create a DeadlineTask subclass that extends Task with a due_date attribute:

  1. Constructor: takes title and due_date (ISO format string like "2026-07-05"). Call super().__init__(title).
  2. __repr__: returns "DeadlineTask('title', 'due_date')".
  3. __str__: shows "📅 title (due: due_date)" plus a "⚠️ OVERDUE!" tag if the date is before today.
  4. __eq__: equal if same title, done, and due_date.
  5. is_overdue() method: returns True if due_date < today.

💡 Use from datetime import date and date.fromisoformat() to parse dates.

⏱ Try for 15 minutes before checking the solution on the next slide.

Give them time to work through this independently. DeadlineTask is a natural extension of PriorityTask that introduces date handling.

✅ Exercise Solution

from datetime import date
 
class DeadlineTask(Task):
    def __init__(self, title, due_date):
        super().__init__(title)
        self.due_date = due_date
 
    def __repr__(self):
        return f"DeadlineTask('{self.title}', '{self.due_date}')"
 
    def __str__(self):
        s = f"📅 {self.title} (due: {self.due_date})"
        if self.is_overdue():
            s += " ⚠️ OVERDUE!"
        return s
 
    def __eq__(self, other):
        if not isinstance(other, DeadlineTask):
            return NotImplemented
        return (super().__eq__(other)
                and self.due_date == other.due_date)
 
    def is_overdue(self):
        return date.fromisoformat(self.due_date) < date.today()
>>> dt = DeadlineTask("Submit report", "2026-07-05")
>>> print(dt)
📅 Submit report (due: 2026-07-05)
>>> dt.is_overdue()
False

Walk through the solution. Highlight how super().__eq__() is reused again. Show how DeadlineTask is only ~15 lines but gets all of Task's functionality for free.

🎯 Key Takeaways

🧬 Inheritance

class Child(Parent): — reuse and extend. Children inherit all methods and attributes from the parent.

⬆️ super()

Call the parent's version of a method. Essential when overriding __init__ so parent attributes are initialized.

🖨️ __str__ & __repr__

__str__ → users (print()). __repr__ → developers (recreates the object). Define __repr__ at minimum.

⚖️ __eq__

Makes == compare by value not identity. Always guard with isinstance() and return NotImplemented.
