🌐 The API Layer

Understanding the OpenWeatherMap API and how to call it from JavaScript.

🔌 Understanding the API

The OpenWeatherMap Current Weather endpoint

Endpoint:

https://api.openweathermap.org/data/2.5/weather?q={city}&appid={API_KEY}&units=metric

Sample response (JSON):

{
  "name": "London",
  "main": {
    "temp": 15.3,
    "feels_like": 14.1,
    "humidity": 72
  },
  "weather": [
    { "main": "Clouds", "description": "overcast clouds", "icon": "04d" }
  ],
  "wind": { "speed": 4.5 }
}
ParameterValueNotes
qCity namee.g. London or Tokyo,JP
appidYour API keyFree tier: 60 calls/min
unitsmetric or imperialMetric = °C, m/s
Walk through the response structure. Point out that `weather` is an array (usually one element). The `icon` code maps to URL `https://openweathermap.org/img/wn/{icon}@2x.png`.

🎣 Step 3: The useWeather Hook

Manage loading, error, and data states for the API call

import { useState } from 'react';

const API_KEY = import.meta.env.VITE_OWM_API_KEY;
const BASE_URL = 'https://api.openweathermap.org/data/2.5/weather';

export function useWeather() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  const fetchWeather = async (city) => {
    setLoading(true);
    setError(null);
    try {
      const res = await fetch(
        `${BASE_URL}?q=${city}&appid=${API_KEY}&units=metric`
      );
      if (!res.ok) throw new Error('City not found');
      const json = await res.json();
      setData(json);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  };

  return { data, loading, error, fetchWeather };
}
🔑 API Key: Create a .env file with VITE_OWM_API_KEY=your_key_here. Vite exposes env vars prefixed with VITE_.
The three-state pattern (loading/error/data) is universal. Show how the finally block ensures loading is always reset. Note that Vite requires the VITE_ prefix for client-side env vars.

⌨️ Step 4: SearchBar Component

A controlled form that sends the city name to the parent

import { useState } from 'react';

export function SearchBar({ onSearch }) {
  const [city, setCity] = useState('');

  const handleSubmit = (e) => {
    e.preventDefault();
    if (city.trim()) onSearch(city.trim());
  };

  return (
    

       setCity(e.target.value)}
        placeholder="Enter city name..."
      />
      
    

  );
}

Key concepts:

  • Controlled input via useState
  • e.preventDefault() stops page reload
  • Lifting state up via onSearch prop
💡 Pattern: The parent (App) passes a callback. SearchBar calls it with the city — it never touches the API directly.
Emphasize that SearchBar is a "dumb" component — it just collects input and calls a prop. All the logic lives in App and useWeather.

🌡️ Step 5: WeatherCard Component

Displays the current conditions from the API response

export function WeatherCard({ data }) {
  if (!data) return null;

  const { name, main, weather, wind } = data;
  const condition = weather[0];

  return (
    <div className="weather-card">
      <h2>{name}</h2>
      <p className="temp">{Math.round(main.temp)}°C</p>
      <p className="desc">{condition.description}</p>
      <div className="details">
        <span>💧 {main.humidity}%</span>
        <span>🌬️ {wind.speed} m/s</span>
        <span>🤔 Feels like {Math.round(main.feels_like)}°C</span>
      </div>
    </div>
  );
}
📐 Destructuring tip: Pulling name, main, weather, wind from data in one line keeps the render clean and readable.
Notice the guard clause: if !data return null. This is a common React pattern for conditional rendering — don't render until the data arrives.

🧩 Bringing It Together

Wire everything up in App.jsx and add the styling.

⚡ Step 7: Wire Up App.jsx

The parent component that connects hook → input → display

import { useWeather } from './hooks/useWeather';
import { SearchBar } from './components/SearchBar';
import { WeatherCard } from './components/WeatherCard';
import { WeatherIcon } from './components/WeatherIcon';

export default function App() {
  const { data, loading, error, fetchWeather } = useWeather();

  return (
    <div className="app">
      <h1>🌤️ Weather Dashboard</h1>
      <SearchBar onSearch={fetchWeather} />

      {loading && <p className="status">Loading...</p>}
      {error && <p className="status error">{error}</p>}

      {data && (
        <div className="result">
          <WeatherIcon code={data.weather[0].icon} />
          <WeatherCard data={data} />
        </div>
      )}
    </div>
  );
}
🧠 Conditional rendering pattern: Only one of loading/error/data shows at a time. The hook guarantees mutual exclusivity via its state machine.
This is the "smart" component that wires everything. The pattern is: one hook call, one SearchBar, and three conditional renders for the three states. Clean and easy to reason about.

🚀 Live Demo: How It Works

End-to-end data flow when the user searches a city

① User types "Tokyo" + clicks Search
        ↓
② SearchBar calls onSearch("Tokyo") → fetchWeather("Tokyo")
        ↓
③ useWeather sets loading=true, calls fetch()
        ↓
④ OpenWeatherMap returns JSON with temperature, humidity, wind
        ↓
⑤ useWeather sets data=json, loading=false
        ↓
⑥ App renders WeatherCard + WeatherIcon with the data

✅ Success path: input → fetch → JSON → render

❌ Error path: city not found → error state → error message

🏋️ Exercise 1: Add Country Flag

Enhance the WeatherCard to show a country flag next to the city name

Task: The API response includes a sys.country field (e.g. "JP" for Japan). Use the emoji flag API to display it alongside the city name.

Hint: Each country code letter maps to a regional indicator emoji.
Formula: String.fromCodePoint(0x1F1E6 + code.charCodeAt(0) - 65) for each letter.

⏱️ Expected output: The city name renders as "🇯🇵 Tokyo" with the flag displayed inline.
This is a creative exercise that teaches them to explore the API response and implement a small utility function. The formula trick is fun — no external library needed for country flags.

🏋️ Exercise 2: Recent Searches

Keep a list of recently searched cities

Task: Track the last 5 cities searched. Display them as clickable buttons so the user can quickly re-query a city.

Hint: Use useState in App to store an array of recent cities. Use the fetchWeather function as the click handler. Cap the array at 5 with .slice(0, 5).

// Structure hint:
const [recent, setRecent] = useState([]);

const handleSearch = (city) => {
  setRecent(prev => [city, ...prev.filter(c => c !== city)].slice(0, 5));
  fetchWeather(city);
};
This teaches array state management and the "deduplicate + cap" pattern. It's a UX feature they'll use on real apps.

🏋️ Exercise 3: Temperature Toggle

Let the user switch between °C and °F

Task: Add a toggle button (switch) that converts the temperature between Celsius and Fahrenheit. The API returns data in whatever units parameter you sent — you'll need a conversion function.

Formula:

C → F: (c * 9/5) + 32
F → C: (f - 32) * 5/9

Hint: Store the unit preference in a useState in App. Pass it down to WeatherCard. Display the converted value using the formula.

💡 Bonus: Persist the unit preference in localStorage so it survives page reloads.
This touches on derived state — the API gives you °C and you compute °F from it rather than making a second API call. Also introduces localStorage for persistence.

📌 Recap

Three things to remember from today.

🎯 Key Takeaways

Write these down — they'll serve you on every API-integrated project.

1. Three-state async pattern
Every API call has exactly three states: loading, error, and data. Always render a UI for all three. Never assume the request will succeed.

2. Separate concerns with custom hooks
Move data-fetching logic into a custom hook. Keep components "dumb" — they receive props and render. This makes testing, reusing, and reasoning about code easier.

3. Lifting state up
App.jsx is the orchestrator. It owns the state, passes callbacks down to SearchBar, and passes data down to WeatherCard. Data flows down, events flow up.

These three patterns (async state management, custom hooks, and unidirectional data flow) are the foundation of every React app they'll work on.
