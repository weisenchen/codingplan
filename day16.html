Why Error Handling?

Without handling

$ python read_data.py
Traceback (most recent call last):
  File "read_data.py", line 3, in <module>
    contents = file.read()
FileNotFoundError: [Errno 2] No such file or directory: 'data.txt'

Program crashes. User has no idea what happened. Bad UX.

With handling

$ python read_data.py
File 'data.txt' not found. Creating a new one.
Enter your data: _

Program recovers gracefully. User gets a helpful message. Good UX.

Key idea: Errors happen — network drops, missing files, bad input. Robust code anticipates them and handles them.

try / except Basics

The fundamental error-handling building block

try / except — Catching an Exception

try:
    # Code that might fail
    number = int(input("Enter a number: "))
    print(f"You entered {number}")
except:
    # Code that runs if anything goes wrong
    print("That wasn't a valid number!")

The recipe:

  • try: Python attempts this block first
  • If an exception occurs, the rest of try is skipped
  • except: catches any exception — use this carefully!
  • If no exception occurs, except is skipped entirely

⚠ Plain except: catches everything (even KeyboardInterrupt). Always prefer specific exceptions.

Catching Specific Exceptions

Different errors deserve different responses. Name the exception type you expect:

try:
    with open("data.txt") as f:
        data = f.read()
except FileNotFoundError:
    print("The file doesn't exist yet — creating it.")
    with open("data.txt", "w") as f:
        f.write("")

Common built-in exceptions:

FileNotFoundError — file doesn't exist

ValueError — wrong value (e.g. int("abc"))

TypeError — operation on wrong type

KeyError — missing dict key

Accessing the Exception Message

Use as to capture the exception object and inspect the error:

try:
    num = int(input("Enter a number: "))
except ValueError as e:
    print(f"Oops! {e}")
    # Output: Oops! invalid literal for int() with base 10: 'hello'

This is useful for:

  • Logging — record the error details for debugging
  • User feedback — show a friendly version of the problem
  • Fallback logic — decide what to do based on the error
# Pattern: log the error, show the user a simple message
import logging

try:
    result = risky_operation()
except Exception as e:
    logging.error(f"Operation failed: {e}")
    print("Something went wrong. Please try again.")

Multiple except Blocks

Handle different errors with different responses:

try:
    filename = input("Enter filename: ")
    multiplier = int(input("Multiplier: "))
    with open(filename) as f:
        value = int(f.read().strip())
    result = value * multiplier
    print(f"Result: {result}")

except FileNotFoundError:
    print(f"'{filename}' not found. Check the name.")
except ValueError:
    print("Couldn't convert to a number. Check your input.")
except Exception as e:
    print(f"Unexpected error: {e}")

Order matters! Python checks except blocks top-to-bottom. Put specific exceptions first, general (Exception) last.

else & finally

Success paths and guaranteed cleanup

else — Runs Only on Success

Code in the else block runs only if no exception was raised. Keeps the try block minimal.

try:
    file = open("settings.json")
except FileNotFoundError:
    print("No settings file. Using defaults.")
    config = {"theme": "dark"}
else:
    # Only runs if open() succeeded
    config = json.load(file)
    file.close()
    print("Settings loaded.")

Why not just put it in try? The else block makes it clear: "this code runs because the risky part succeeded." It also avoids accidentally catching exceptions that shouldn't be caught.

finally — Runs No Matter What

finally always runs — whether the try succeeded, failed, or even if you return from inside try. Use it for cleanup.

try:
    file = open("data.txt")
    data = file.read()
    return process(data)
except FileNotFoundError:
    print("File not found.")
    return []
finally:
    # ALWAYS runs — even if we returned from try or except!
    file.close()

Common finally use cases:

  • Closing files, network connections, or database handles
  • Releasing locks or temporary resources
  • Logging that an operation completed (success or failure)

💡 With file operations, the with statement handles finally automatically — but for custom resources, finally is essential.

The Complete Pattern

All four clauses together — the full error-handling toolkit:

try:
    # Risky code
    num = int(input("Number: "))
    result = 100 / num
except ValueError:
    print("That's not a number!")
except ZeroDivisionError:
    print("Can't divide by zero!")
else:
    # Only if no exception
    print(f"Result: {result}")
finally:
    # Always runs
    print("Operation attempted.")
# Example runs:

# Input: "hello"
That's not a number!
Operation attempted.

# Input: "5"
Result: 20.0
Operation attempted.

# Input: "0"
Can't divide by zero!
Operation attempted.

Walkthrough: Safe File Reader

Write a function that reads a number from a file safely — handling every failure point:

def read_number(filename):
    """Read a number from file. Returns None on failure."""
    try:
        with open(filename) as f:
            content = f.read().strip()
        if not content:
            return 0
        return int(content)
    except FileNotFoundError:
        print(f"'{filename}' not found. Creating with value 0.")
        with open(filename, "w") as f:
            f.write("0")
        return 0
    except ValueError:
        print(f"'{filename}' doesn't contain a valid number.")
        return None

Each error gets a specific, helpful response. The function never crashes — it always returns a usable value.

Raising Exceptions with raise

Sometimes you need to signal that something's wrong. Use raise to throw your own exceptions:

def set_age(age):
    if age < 0:
        raise ValueError("Age can't be negative!")
    if age > 150:
        raise ValueError("Age seems unrealistic.")
    print(f"Age set to {age}")

Bad practice

def divide(a, b):
    if b == 0:
        return "Error!"  # mixed types!
    return a / b

Returning different types is confusing. Don't do this.

Good practice

def divide(a, b):
    if b == 0:
        raise ValueError("Division by zero")
    return a / b

Raise an exception. Let the caller decide how to handle it. ✓

Key Takeaways

1

try/except catches errors — wrap risky code in try, handle failures in except. Always catch specific exceptions.

2

else runs on success, finally always runs — else is for "if it worked", finally is for cleanup (close files, release resources).

3

Raise, don't return errors — use raise ValueError(...) instead of returning mixed types. Let the caller handle it.

4

Robust code anticipates failure — file not found, bad input, network issues. Handle each case specifically for the best user experience.
