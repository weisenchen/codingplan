🧠 Why Git?

The Problem

  • ❌ mycode_final_v2_really_final.py
  • ❌ "Oops, I deleted something and can't undo"
  • ❌ "Which version fixed the bug last week?"
  • ❌ No collaboration without file sharing chaos

Git Is the Solution

  • ✅ Time machine — every commit is a save point
  • ✅ Safety net — experiment fearlessly, revert if needed
  • ✅ History — see what changed, when, and who
  • ✅ Collaboration — multiple people on the same code
💡 Analogy: Git is like save-states in a video game. Before a boss fight (big change), you save. If you die (break something), you load the last save. Git commits are those save files.
Use the video game analogy to make it click. Emphasise that Git stores snapshots, not just file copies. The staging area is what makes Git different from older VCS tools like SVN.

🔄 The Three States of Git

Every file in a Git repo lives in one of three states:

1. Modified
You changed a file but haven't told Git about it yet. It lives in your working directory.
2. Staged
You marked the changes for the next snapshot. It's in the staging area (index), ready to be committed.
3. Committed
The snapshot is saved in Git's database. You can return to this state forever.
Working Directory → git add → Staging Area → git commit → Repository
This three-state model is the most important mental model in Git. Emphasise that staging lets you craft exactly what goes into each commit — you can change 3 files but only commit 2 of them. This is unique to Git.

🚀 Step 1: git init & git status

Turn any folder into a Git repository and check what's happening inside.

# Create a project folder
mkdir my-first-repo
cd my-first-repo

# Initialize a new Git repository
git init
# → Initialized empty Git repository in .../my-first-repo/.git/

# Check the state of your repo
git status
# → On branch master
# → No commits yet
# → nothing to commit

# List the hidden .git folder (Git's engine room)
ls -la .git/
💡 git init creates a .git/ subdirectory — this is where all your history lives. Never delete or edit it manually!
Run these commands live. Show that .git/ is just a folder — Git's database. After init, the repo is empty: no commits yet, nothing staged. `git status` becomes your most-used Git command.

📦 Step 2: git add — The Staging Area

Create a file, then stage it for the next commit.

# Create a file
echo "# My First Repo" > README.md

# Git sees an untracked file
git status
# → Untracked files:
# →   README.md

# Stage the file (add to staging area)
git add README.md

# Check again
git status
# → Changes to be committed:
# →   new file:   README.md

# Stage everything at once
git add .
💡 git add . stages all changes in the current directory. Use git add <file> for precision.
💡 You can stage parts of a file with git add -p (advanced — coming later).
The staging area is Git's killer feature. Show that you can stage multiple files incrementally. The `git add .` shorthand is common but warn learners to check status first so they don't accidentally stage trash files.

💾 Step 3: git commit — Save the Snapshot

Commit the staged changes with a message describing what you did.

# Make another file
echo "print('Hello Git!')" > hello.py
git add hello.py

# Commit with a message
git commit -m "Add README and hello script"

# → [master (root-commit) a1b2c3d] Add README and hello script
# →  2 files changed, 2 insertions(+)
# →  create mode 100644 README.md
# →  create mode 100644 hello.py

# Check status — clean!
git status
# → nothing to commit, working tree clean
💡 Best practice: Write commit messages in imperative present tense — "Add feature", not "Added feature" or "Adding feature". The -m flag is for short messages. For longer ones, omit -m to open an editor.
Walk through the output: commit hash (a1b2c3d), branch name (master), files changed. The hash is a unique identifier for this snapshot. Good commit messages are a differentiator on co-op — reviewers read commit history.

📜 Step 4: git log — Read the History

See every commit you've made, in reverse chronological order.

# Make another change and commit it
echo "def greet(name):\n    return f'Hello, {name}'" >> hello.py
git add hello.py
git commit -m "Add greet function"

# View commit history
git log
# → commit 7e8f9a0 (HEAD -> master)
# → Author: Your Name <you@example.com>
# → Date:   Tue Jun 2 10:00:00 2026 -0400
# →
# →     Add greet function
# →
# → commit a1b2c3d
# → Author: Your Name <you@example.com>
# → Date:   Tue Jun 2 09:55:00 2026 -0400
# →
# →     Add README and hello script

# Compact one-line view
git log --oneline
# → 7e8f9a0 Add greet function
# → a1b2c3d Add README and hello script
💡 git log --oneline is your everyday view. git log --graph visualises branches (Week 4).
💡 Press q to exit the log pager. Use git log --oneline -5 to see only the last 5 commits.
Git log is your project diary. Show --oneline and --graph (even though we haven't done branches yet, learners should know it exists). The hash is a SHA-1 checksum — you don't need to remember it, just copy-paste.

📋 Git Cheat Sheet — Today's Commands

Command What It Does
git init Create a new Git repository in the current folder
git status Show which files are modified, staged, or untracked
git add <file> Stage a specific file for the next commit
git add . Stage all changes in the current directory
git commit -m "msg" Commit staged changes with a message
git log Show commit history in detail
git log --oneline Show commit history as a compact list
💡 Pro tip: Run git status constantly — it's your compass. After every git add and git commit, run git status to confirm the state. It's impossible to break anything with these commands!
Reference slide learners can come back to. The most important habit: git status after every command until it becomes automatic.

🏋️ Exercise: Own Your First Repo

🎯 Task

Create a Git repository for a recipe collector, making at least 3 commits with meaningful messages.

  1. mkdir recipe-collector && cd recipe-collector
  2. git init
  3. Create recipes/pasta.md with "# Spaghetti Carbonara"
  4. git add . then git status
  5. git commit -m "Add pasta recipe stub"
  6. Edit pasta.md — add ingredients list (3+ items)
  7. git add -u (stage modified files only) then commit
  8. Create recipes/salad.md with a recipe title
  9. Add and commit it
  10. Run git log --oneline — you should see 3 commits
💡 Stuck? Use mkdir -p recipes to create the subfolder. git add -u stages modified files (already tracked), while git add . stages everything including new files.
The exercise is deliberately about food — low cognitive load on the content so the learner can focus on the Git mechanics. Watch for the `git add -u` vs `git add .` distinction.

🎯 Key Takeaways

📌 Remember This

  • 📌 Working → Staged → Committed. Changes flow left to right through three states. git add moves left, git commit moves right.
  • 📌 Commit often, with good messages. Small focused commits make history readable. Imperative tense ("Add", not "Added").
  • 📌 git status is your compass. When in doubt, run it. It tells you exactly what's happening and what to do next.
