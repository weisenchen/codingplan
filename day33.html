🔗 Why Split Data Into Tables?

Imagine storing orders and the customer's name in every row:

-- Bad: repeated data, update nightmare
CREATE TABLE orders_bad (
  order_id INT,
  item      TEXT,
  customer  TEXT,   -- "Alice Wong" repeated 20x
  email     TEXT    -- same email repeated 20x
);

Problems: Redundant data, inconsistent updates, wastes space.

Instead, we normalize: store each fact once, link via IDs.

-- Good: one customer record, many order references
CREATE TABLE customers (
  customer_id INT PRIMARY KEY,
  name        TEXT,
  email       TEXT
);
CREATE TABLE orders (
  order_id    INT PRIMARY KEY,
  customer_id INT REFERENCES customers(customer_id),
  item        TEXT
);

🔷 INNER JOIN — Matches in Both Tables

INNER JOIN returns rows only when there's a match in both tables.

Analogy: Like a Venn diagram intersection — only the overlapping part.
-- Get each user + their order info (only users who've ordered)
SELECT users.name, orders.order_id, orders.quantity
FROM users
INNER JOIN orders ON users.user_id = orders.user_id;

-- Result (hypothetical):
--  name      | order_id | quantity
--  ----------+----------+---------
--  Alice     | 101      | 2
--  Alice     | 102      | 1
--  Bob       | 103      | 5
--  Diana     | 104      | 1
--  (users with no orders are excluded)
5 / 12
5 / 12
Emphasize: users who never placed an order simply won't appear. The JOIN condition is the key — "ON users.user_id = orders.user_id" means "match rows where these columns are equal." Use your hands to show two overlapping circles.

🔷 LEFT JOIN — Keep All Left-Side Rows

LEFT JOIN returns all rows from the left table, plus matches from the right. Unmatched right-side columns are NULL.

Analogy: "Left table is the guest list — everyone on the list shows up. If someone didn't bring a plus-one, that spot is empty (NULL)."
-- All users, with their orders (NULL if none)
SELECT users.name, orders.order_id, orders.quantity
FROM users
LEFT JOIN orders ON users.user_id = orders.user_id;

-- Result (hypothetical):
--  name      | order_id | quantity
--  ----------+----------+---------
--  Alice     | 101      | 2
--  Alice     | 102      | 1
--  Bob       | 103      | 5
--  Charlie   | NULL     | NULL     ← no orders
--  Diana     | 104      | 1
--  Eve       | NULL     | NULL     ← no orders
6 / 12
6 / 12
Draw the Venn — LEFT JOIN is the entire left circle. Use Charlie as the example: he's on the user list but has no orders. In the result, order columns show NULL. Ask "when would you use LEFT JOIN over INNER JOIN?" (When you need all users, even inactive ones.)

⚖️ INNER vs LEFT — Which One to Use?

✅ INNER JOIN

Use when you only want rows with matches:

  • Active customers who've ordered
  • Orders that have a valid product
  • Students enrolled in a course
  • Anything where NULL = irrelevant

✅ LEFT JOIN

Use when you need all left-side rows:

  • All users, even inactive ones
  • All products, even unsold ones
  • Report with "0" or NULL for missing data
  • Finding orphans (WHERE right IS NULL)
Pro tip: Use WHERE orders.order_id IS NULL with a LEFT JOIN to find users without orders — a very common pattern.

🔗 Joining Three Tables

Chain JOINs to connect across multiple tables. Each JOIN adds one more table:

-- Users → Orders → Products: what did each user order, and what's it called?
SELECT
  users.name,
  orders.order_id,
  products.name AS product_name,
  products.price,
  orders.quantity,
  (products.price * orders.quantity) AS total
FROM users
INNER JOIN orders   ON users.user_id     = orders.user_id
INNER JOIN products ON orders.product_id = products.product_id
ORDER BY total DESC;
Read it like a sentence: "From users, join orders where user_id matches, then join products where product_id matches."

8 / 12
This is the payoff — the three-table join. Read the query aloud as a sentence. Point out the alias AS and the computed total column. This is a real reporting query.

📝 Table Aliases — Cleaner Queries

Type less, read more. Use AS (or just a space) to alias table names:

-- Without aliases (verbose):
SELECT users.name, orders.order_id, products.name, products.price
FROM users
INNER JOIN orders ON users.user_id = orders.user_id
INNER JOIN products ON orders.product_id = products.product_id;

-- With aliases (clean):
SELECT u.name, o.order_id, p.name AS product, p.price
FROM users u
INNER JOIN orders o   ON u.user_id = o.user_id
INNER JOIN products p ON o.product_id = p.product_id;
Pitfall: If two tables have the same column name (e.g. name in both users and products), you must prefix with table name or alias: u.name, p.name.

⚠️ Common JOIN Pitfalls

Pitfall 1: Missing JOIN condition
FROM users JOIN orders (without ON) → Cartesian product — every user paired with every order. Catastrophic with large tables.
Pitfall 2: Duplicate rows
If a user has 3 orders and you join to orders then to a table with multiple rows per order, rows multiply. Always check cardinality before joining.
Pitfall 3: NULL in JOIN condition
ON a.col = b.col — if either value is NULL, the row does not match. NULL ≠ NULL in SQL.
Debug quickly: Run SELECT COUNT(*) before and after each JOIN to see if rows exploded.
10 / 12
10 / 12
These are the three mistakes everyone makes. Missing ON is the most destructive — it's a silent cartesian product. I've seen interns bring down a production DB this way. Always, always write the ON clause.

🎯 Key Takeaways

  • INNER JOIN — only matching rows from both tables. Use when NULLs don't make sense.
  • LEFT JOIN — all rows from the left table, matches from the right (or NULL). Use when you need everything from one side.
  • Foreign keys are the glue — they connect tables and define the JOIN condition.
  • Always write the ON clause — missing it causes a cartesian explosion.
