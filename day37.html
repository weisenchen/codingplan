🧠 What is Object-Oriented Programming?

OOP is a way of organising code around objects — bundles of data and behaviour that model real-world things.

❌ Without OOP

  • Scattered variables: student_name, student_age, student_grades
  • Separate functions: print_student(), calc_average()
  • Easy to lose connection between data and the functions that use it

✅ With OOP

  • Everything in one place: a Student class
  • Data = attributes (.name, .age)
  • Behaviour = methods (.print_summary(), .average_grade())
  • Reusable: create many students from one class
🏭

Analogy: A class is a blueprint / factory mould. An object is the actual thing the mould produces. One blueprint → many objects.

Start with the big picture. OOP is about grouping related data and functions together. Use the blueprint/object analogy: the blueprint for a house (class) vs. actual houses (objects). Each house has its own address, colour, occupants — but they all share the same structure defined in the blueprint.

🏗️ The class Keyword

A class is defined with class ClassName:. Inside, __init__ is the constructor that runs when you create a new object.

class Student:
        def __init__(self, name, age):
            self.name = name
            self.age = age
            self.grades = []
    # Creating an object (instance)
    alice = Student("Alice", 20)
    print(alice.name)  # Alice

class

Declares the blueprint. Convention: PascalCase (Student, TaskManager).

__init__

Constructor method. Called once when the object is created. Sets up initial state.

self

The object itself. Every method gets self as the first parameter so you can access attributes.

Walk through the syntax carefully. Emphasise three things: (1) PascalCase naming convention for classes, (2) __init__ runs automatically on creation, (3) self is always the first parameter and refers to the specific instance. Use the whiteboard: draw two Student objects, each with their own .name and .age in memory, but sharing the same class definition.

📦 Attributes: Instance vs. Class

Attributes store data on an object. There are two kinds:

class Student:
        school = "Waterloo Collegiate"  # Class attribute — shared by ALL students
        
        def __init__(self, name, age):
            self.name = name              # Instance attribute — unique per student
            self.age = age
            self.grades = []
    
    print(alice.school)  # "Waterloo Collegiate" — accessed via instance
    print(Student.school) # "Waterloo Collegiate" — accessed via class

📍 Instance Attributes

  • Set in __init__ via self.attr = value
  • Each object has its own copy
  • alice.name ≠ bob.name

🌍 Class Attributes

  • Set directly on the class body
  • Shared by all instances
  • Good for defaults, constants, config
This is a common interview question: "What's the difference between class and instance attributes?" Instance = each object's own data. Class = shared across all objects. Demo with two Student objects: change alice.school and show it doesn't affect the class attribute if you assign via the instance (it shadows it). Show Student.school stays the same.

🏃 Methods — Behaviour on Objects

Methods are functions that belong to a class. They operate on the object's data via self.

class Student:
        school = "Waterloo Collegiate"
        
        def __init__(self, name, age):
            self.name = name
            self.age = age
            self.grades = []
    
        def add_grade(self, grade):       # Method: adds a grade
            self.grades.append(grade)
        
        def average(self):                 # Method: computes average
            if not self.grades:
                return 0
            return sum(self.grades) / len(self.grades)
    
        def is_passing(self):               # Method: returns bool
            return self.average() >= 60
    
    alice = Student("Alice", 20)
    alice.add_grade(85)
    alice.add_grade(92)
    print(alice.average())     # 88.5
    print(alice.is_passing())  # True

Key insight: Every method takes self first. When you call alice.average(), Python automatically passes alice as self.

Methods are just functions attached to a class. The magic is that Python passes the instance as the first argument automatically. Demonstrate: Student.average(alice) is equivalent to alice.average(). This is a common interview question / whiteboard trap — make sure learners understand it.

📝 Walkthrough: Task Class (Project 1 Flashback)

Remember Project 1 — the CLI Todo List? Let's model its core with a Task class.

1

Define Task with title, priority (default: "medium"), and completed (default: False).

class Task:
        def __init__(self, title, priority="medium"):
            self.title = title
            self.priority = priority
            self.completed = False
    
        def mark_done(self):
            self.completed = True
            return f"{self.title} completed!"
    
        def __str__(self):
            status = "✓" if self.completed else "○"
            return f"{status} [{self.priority}] {self.title}"
2

Test it — create tasks, mark one done, print them.

t1 = Task("Apply to 5 co-op jobs", "high")
    t2 = Task("Review data structures")
    print(t1)
    print(t2)
    t1.mark_done()
    print(t1)
○ [high] Apply to 5 co-op jobs
○ [medium] Review data structures
✓ [high] Apply to 5 co-op jobs

Bonus: __str__ is a magic method — it defines how print() displays your object. More magic methods in a future lesson.

🔧 Working with Objects — Common Patterns

📋 Creating objects

alice = Student("Alice", 2)
bob = Student("Bob", 3)
tasks = [
    Task("Study"),
    Task("Apply", "high")
]

🔄 Modifying objects

alice.year = 3          # Direct attribute
alice.enroll("CS 350") # Via method
tasks[0].mark_done()     # From list
tasks[1].priority = "low"

🔍 Inspecting objects

print(alice.info())
print(type(alice))     # <class '__main__.Student'>
print(isinstance(alice, Student))  # True

📂 Collections of objects

for t in tasks:
    if not t.completed:
        print(t)
# Filtering by attribute
high = [t for t in tasks
        if t.priority == "high"]

✅ Exercise Solutions

📖 Book class

class Book:
    def __init__(self, title, author, pages):
        self.title = title
        self.author = author
        self.pages = pages
        self.current_page = 0

    def read(self, pages):
        self.current_page += pages
        if self.current_page > self.pages:
            self.current_page = self.pages
        return f"Read {self.current_page} of {self.pages} pages ({self.progress()})"

    def progress(self):
        pct = (self.current_page / self.pages) * 100
        return f"{pct:.0f}%"

🏦 BankAccount class

class BankAccount:
    interest_rate = 0.02

    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance

    def deposit(self, amount):
        self.balance += amount
        return f"Deposited ${amount}. Balance: ${self.balance}"

    def withdraw(self, amount):
        if amount > self.balance:
            return "Insufficient funds!"
        self.balance -= amount
        return f"Withdrew ${amount}. Balance: ${self.balance}"

    def apply_interest(self):
        self.balance += self.balance * BankAccount.interest_rate
Review both solutions. For Book, note the clamping logic (can't read past the last page). For BankAccount, highlight: (1) using `BankAccount.interest_rate` vs `self.interest_rate` to access class attribute, (2) the validation in withdraw(). Mention that real banking apps use Decimal, not float, for money.

⚠️ Common Pitfalls & How to Avoid Them

❌ Forgetting self

# BAD — missing self
def __init__(name):       # TypeError!
    self.name = name
# GOOD
def __init__(self, name):

Python passes the instance as first arg. Always list self first in method signatures.

❌ Mutable default args

# BAD — shared across instances!
def __init__(self, grades=[]):
# GOOD — None + create inside
def __init__(self, grades=None):
    self.grades = grades or []

Default args are evaluated once at definition. Mutable defaults cause bugs.

❌ Accessing wrong scope

# BAD — local variable hides self.name
name = "Class name"
def info(self):
    return name  # Uses class-level, not instance!

Always use self.attr to access instance data from within methods.

❌ Mutable class attributes

# Students all share this list!
class Student:
    grades = []  # Class attribute

Class attributes should be immutable (strings, numbers, tuples) or constants.

Golden rule: If it varies by instance, put it in __init__ with self. If it's shared / constant, use a class attribute.

These are the top 4 mistakes beginners make with OOP in Python. The mutable default arg trap is famously asked in Python interviews. The mutable class attr bug is subtle — show a demo where two students' grades get mixed up.

📌 Key Takeaways

1. Class = blueprint, Object = instance

class Student: defines the blueprint. alice = Student("Alice", 2) produces one object. You can make many objects from one class.

2. __init__ + self set up instance data

__init__ runs on creation. self.attr stores data on the specific object. Every method must take self as the first parameter.

3. Instance attrs are per-object; class attrs are shared

Instance attrs in __init__ (unique), class attrs on the class body (shared). Mutable class attrs are dangerous.

4. Methods bundle behaviour with data

A method is a function on a class. It operates on self to read/write the object's data. Call with object.method().
