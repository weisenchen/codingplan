Why Test Your Code?

  • Catch regressions — a change in one place breaks something that used to work
  • Document intent — tests show what each function is supposed to do
  • Refactor with confidence — tests are your safety net when restructuring code
  • Save time — a 5-minute test catches a 30-minute debugging session
  • Impress interviewers — testing is expected in professional codebases

⏳ pytest Fundamentals

Step 1 — Install & First Test

pytest is a third-party library. Install it with pip:

pip install pytest

Check the version:

pytest --version

Now create a file called test_first.py:

# test_first.py

def test_addition():
    assert 1 + 1 == 2

def test_string():
    assert "hello".upper() == "HELLO"

def test_failing_intro():
    # This will fail — we'll learn to read failure output
    assert 2 * 3 == 7

Step 2 — Run & Read Output

Run all tests in a file:

pytest test_first.py

Or let pytest discover tests automatically:

pytest

You'll see output like:

..F
FAILED test_first.py::test_failing_intro - AssertionError: assert (2 * 3) == 7

2 passed, 1 failed in 0.12s

KEY A dot . = pass, F = fail. pytest tells you exactly which assertion failed and what the actual value was.

The assert Statement

Python's built-in assert is all pytest needs. No special assertions API.

Equality

assert result == expected

Inequality

assert result != bad_value

Containment

assert "sub" in text
assert key in dictionary

Truthiness

assert result        # truthy
assert not result    # falsy
assert result is None

Approximate (floats)

assert abs(a - b) < 1e-6

Exceptions

import pytest

def test_raises():
    with pytest.raises(ValueError):
        int("abc")

🧪 Test Discovery

How pytest Discovers Tests

Run pytest with no arguments and it searches the current directory:

File naming

  • test_*.py
  • *_test.py

Function naming

  • def test_*():

Class naming

  • class Test*:
  • No __init__ needed

Selective runs

# Run a specific file
pytest test_core.py

# Run a specific test
pytest test_core.py::test_add

# Run tests matching a keyword
pytest -k "user"

# Stop after first failure
pytest -x

# More verbose
pytest -v

Walkthrough — Testing a Calculator

Create calc.py:

# calc.py

def add(a, b):
    return a + b

def subtract(a, b):
    return a - b

def multiply(a, b):
    return a * b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b

Now create test_calc.py — see next slide →

Walkthrough — test_calc.py

# test_calc.py
from calc import add, subtract, multiply, divide
import pytest

def test_add():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0
    assert add(0, 0) == 0

def test_subtract():
    assert subtract(5, 3) == 2
    assert subtract(0, 1) == -1

def test_multiply():
    assert multiply(3, 4) == 12
    assert multiply(-2, 3) == -6

def test_divide():
    assert divide(10, 2) == 5
    assert divide(7, 2) == 3.5

def test_divide_by_zero():
    with pytest.raises(ValueError):
        divide(10, 0)

Run: pytest test_calc.py -v

🔄 Test-Driven Development

The TDD Cycle: Red → Green → Refactor

🔴 RED

Write a failing test first. Define what the function should do, even though it doesn't exist yet.

🟢 GREEN

Write the minimum code to make the test pass. No extras, no premature optimization.

🔵 REFACTOR

Clean up the code. Tests pass? You can refactor safely — the tests will catch regressions.

TDD forces you to think about requirements before implementation. This produces cleaner, more testable code.

TDD Demo — is_palindrome

🔴 Step 1 — Write the test first

# test_strings.py
from strings import is_palindrome

def test_palindrome():
    assert is_palindrome("racecar")
    assert is_palindrome("level")
    assert is_palindrome("a")

def test_non_palindrome():
    assert not is_palindrome("hello")
    assert not is_palindrome("abc")

def test_cased():
    assert is_palindrome("Racecar")

🟢 Step 2 — Write the code

# strings.py
def is_palindrome(s):
    normalized = s.lower()
    return normalized == normalized[::-1]

Run: pytest test_strings.py

Exercise 1 — Test a get_grade Function

📝 Task

Given this function in grades.py, write tests for it:

def get_grade(score):
    if score < 0 or score > 100:
        raise ValueError("Score must be 0-100")
    if score >= 90: return "A"
    if score >= 80: return "B"
    if score >= 70: return "C"
    if score >= 60: return "D"
    return "F"

💡 Test boundaries: 0, 59, 60, 69, 70, 79, 80, 89, 90, 100, and invalid values.

Exercise 1 — Solution

# test_grades.py
from grades import get_grade
import pytest

def test_grade_a():
    assert get_grade(95) == "A"
    assert get_grade(90) == "A"

def test_grade_b():
    assert get_grade(85) == "B"
    assert get_grade(80) == "B"

def test_grade_c():
    assert get_grade(75) == "C"

def test_grade_d():
    assert get_grade(65) == "D"
    assert get_grade(60) == "D"

def test_grade_f():
    assert get_grade(59) == "F"
    assert get_grade(0) == "F"

def test_invalid():
    with pytest.raises(ValueError):
        get_grade(-1)
    with pytest.raises(ValueError):
        get_grade(101)

Run: pytest test_grades.py -v

Exercise 2 — TDD: Fibonacci

📝 Task — Use TDD

Write tests first for a fib(n) function that returns the n-th Fibonacci number.

Sequence: fib(0)=0, fib(1)=1, fib(2)=1, fib(3)=2, fib(4)=3, fib(5)=5, fib(6)=8

Then implement fib(n) in fib.py.

💡 Edge case: fib(0) should return 0. fib(1) should return 1.

TDD REMINDER Write test_fib.py first. Run it → it fails (RED). Then write fib.py → passes (GREEN). Refactor if needed.

Exercise 2 — Solution

test_fib.py (write first)

from fib import fib

def test_fib_zero():
    assert fib(0) == 0

def test_fib_one():
    assert fib(1) == 1

def test_fib_small():
    assert fib(2) == 1
    assert fib(3) == 2
    assert fib(4) == 3
    assert fib(5) == 5
    assert fib(6) == 8

def test_fib_larger():
    assert fib(10) == 55
    assert fib(20) == 6765

fib.py (write second)

def fib(n):
    if n < 0:
        raise ValueError("n must be >= 0")
    if n == 0:
        return 0
    if n == 1:
        return 1
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b

All 6 tests pass. TDD complete! ✅

Exercise 3 — Test a TodoList Class

📝 Task

Test this class from todo.py — think about edge cases:

class TodoList:
    def __init__(self):
        self.tasks = []

    def add(self, task):
        self.tasks.append(task)

    def remove(self, task):
        if task not in self.tasks:
            raise KeyError(f"Task '{task}' not found")
        self.tasks.remove(task)

    def list_all(self):
        return self.tasks.copy()

    def is_done(self):
        return len(self.tasks) == 0

💡 Test: adding tasks, removing tasks, removing a non-existent task, listing gives copies, empty state

Exercise 3 — Solution

# test_todo.py
from todo import TodoList
import pytest

def test_add_and_list():
    t = TodoList()
    t.add("buy milk")
    t.add("write tests")
    assert t.list_all() == ["buy milk", "write tests"]

def test_remove_task():
    t = TodoList()
    t.add("learn pytest")
    t.remove("learn pytest")
    assert t.is_done()

def test_remove_missing():
    t = TodoList()
    with pytest.raises(KeyError):
        t.remove("nonexistent")

def test_list_isolation():
    t = TodoList()
    t.add("task")
    lst = t.list_all()
    lst.append("hack")
    assert t.list_all() == ["task"]

def test_empty():
    t = TodoList()
    assert t.is_done()
    assert t.list_all() == []

Run: pytest test_todo.py -v

Key Takeaways

1
pytest uses plain assert — no special assertion methods. Install with pip install pytest and run with pytest.
2
Test discovery is convention-based — name files test_*.py, functions def test_*():. Use -k and -x flags for selective runs.
3
TDD: Red → Green → Refactor — write a failing test first, then the minimum code to pass it, then clean up.
4
Test exceptions with pytest.raises — and always test edge cases: boundaries, emptiness, errors.
